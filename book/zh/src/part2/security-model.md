# 安全模型

> **定位**：本章把散落在前几章的安全机制收拢成五条原则，并给出「威胁 → 防线」对照。前置依赖：第 3 章、第 5.4 章。评估者可与前言路径 B 配合阅读。

HAgency 给了 Agent 很大的行动自由，这份自由必须配上同样坚固的边界。

## 五条原则

**1. Robrix2 永远不是授权来源。** Robrix2 只做两件事：展示（审批卡片、工作流状态）与发起（把你的点击变成结构化 Matrix 事件）。所有授权判定发生在 agent-chat 服务端：verdict 的真实发送者（`event.sender`）必须等于绑定的 owner 账号 —— 不信任 display name，不信任 payload 里自称的身份；房间、agent、project、request_id、input_digest 逐项吻合；审批绑定字段只从**原始事件**读取，`m.replace` 编辑无法篡改一张已发出的卡。即使客户端被替换或伪造，服务端校验依然成立。

**2. 审批一次性、有时效、防重放。** `Approve once` 顾名思义 —— 每张卡只放行一次执行；服务端先「消费」审批再通知运行时，allow 永不重放。默认 5 分钟过期，过期点击在客户端与服务端双重拒绝。`input_digest`（对请求规范化内容 —— 含工具名与命令输入预览 —— 的 SHA-256）把授权钉死在**这一次请求**上：预览范围内换一个字符都对不上号（预览截断在 8KB，覆盖绝大多数真实命令）。

**3. Fail-closed：一切异常等于拒绝。** 从 Codex 的审批 hook 到 Claude 的权限通道，链路上任何一环异常 —— 超时、解析失败、通道不可用、完整性校验不过 —— 运行时收到的都是明确的 **deny**，而不是静默放行或无限等待。Codex 审批 hook 绑定脚本 SHA-256 并自校验完整性；hook 首次启用需在本地终端显式输入 `TRUST` 确认。

**4. 加密通道与密钥卫生。** 审批房间强制端到端加密（Megolm），审批详情对 homeserver 不可见。Robrix2 发送 verdict 前显式刷新桥的设备密钥并轮换出站房间密钥，确保桥的当前设备一定能解密；桥对暂时解不开的 verdict 持久化排队等待房间密钥，而不是丢弃或误判。

**5. 受管的运行时。** Agent 的编码运行时以受管方式启动：Claude Code 使用 `--permission-mode auto` + 审批通道，敏感命令（`gh *`、`git push *`）配置为必须请示；Codex 使用 `workspace-write` 沙箱 + `on-request` 审批。tmux 会话由受管脚本启动并打上标记 —— 手工重开一个绕过审批参数的「野生」实例会被启动器拒绝。

## 威胁 → 防线对照

| 威胁 | 防线 | 出处 |
|------|------|------|
| 有人在群里冒充 owner 说「同意」 | 文字回复不是审批；verdict 的 `event.sender` 服务端校验 | 原则 1 |
| 重放一张旧的批准 | 单次消费 + TTL + request_id 绑定 | 原则 2 |
| 批准 A 命令、实际执行 B 命令 | `input_digest` 内容级绑定 | 原则 2 |
| 审批链路故障导致「默认放行」 | 全链路 fail-closed，异常一律 deny | 原则 3 |
| homeserver 或网络窥探审批内容 | 审批房 E2EE，服务器只见密文 | 原则 4 |
| 篡改审批 hook / 绕过受管启动 | hook SHA-256 自校验 + TRUST 信任确认 + 受管 PID 标记 | 原则 3 / 5 |
| 编辑（m.replace）已发出的审批卡 | 绑定字段只读原始事件 | 原则 1 |

**边界声明**：这套模型防的是「Agent 越权执行」与「授权通道被伪造/重放/旁路」。它不防 owner 自己批错（所以卡片必须把命令预览给你看清楚），也不防你机器上的 root 级攻击者（那已经越过了任何应用层安全的假设）。

---

**Agent 的能力可以外包，人的授权不能外包。** HAgency 用协议设计和密码学，把这句话变成系统性质，而非美好愿望。
