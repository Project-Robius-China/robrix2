# Agent-to-App Simplified Design

Status: proposed replacement direction for the current Robrix2 agent-to-app design.

This document intentionally narrows the Robrix2 design to what the current code can support cleanly. The aichat example proves a useful runtime pattern, but Robrix2 should not copy it directly: aichat owns the local agent loop and host state, while Robrix2 renders Matrix events produced elsewhere.

Product direction built on this model: [`agent-mission-room-design.md`](agent-mission-room-design.md).

## What We Keep

Robrix2 already has a working event-to-Splash path:

```text
Matrix event content
  -> org.octos.app envelope
  -> app_registry::render_app_envelope_to_splash()
  -> AppFactory::init(initial_state)
  -> RenderedApp::render(language)
  -> SplashHost loads static .splash template
  -> $state.* binding
  -> RoomScreen sets content.splash_card text
```

The current entry point is `src/home/room_screen.rs`: `org.octos.app` is read from the original event content, rendered through `app_registry`, and wins over the legacy raw `org.octos.splash_card` path. This is the right Robrix2 foundation because Matrix event history remains the source of truth.

The useful current pieces are:

- `org.octos.app` envelope parsing.
- Static `AppFactory` registry.
- Weather and news as built-in capabilities.
- Static `.splash` templates under `src/home/app_registry/templates/`.
- `$state.path` replacement before the Splash widget receives text.
- Defensive fallback to Matrix `body`.
- Basic template preflight: widget references, local function references, state binding paths, and host-owned attribution fields.

## What We Do Not Keep In V1

The current design is too broad because it describes the long-term system as if it were the next implementation unit. These should move out of the v1 design:

- Five-layer architecture as the primary explanation.
- Seven whitelists as the primary safety model.
- Resolver LLM and Template-Author LLM split.
- LLM-generated templates and repair loops.
- Runtime dynamic capability discovery.
- Plugin or marketplace distribution.
- Stateful mini-app host runtime.
- Path-scoped incremental updates beyond static render.
- In-card action routing as part of the card renderer.

Some of these may be future work, but putting them in the main design makes the current system harder to reason about and harder to implement safely.

## V1 Contract

The shipped V1 render path is static and event-sourced. The first interactive
step is `agent2view`: a rendered app may have a local view session scoped by the
app envelope, while the Matrix event remains the source of the initial state.

An agent may send a Matrix event with:

```json
{
  "body": "Plain text fallback",
  "org.octos.app": {
    "type": "weather",
    "version": 2,
    "scope": "message",
    "app_id": "optional-for-message-scope",
    "initial_state": {}
  }
}
```

Robrix2 may render the app only if:

- `type` is registered in the local `app_registry`.
- `version` is supported by that app factory.
- `initial_state` validates.
- the factory can build a template view model.
- the selected static template exists.
- the template passes host preflight.
- every `$state.path` in the template can be resolved.

If any step fails, Robrix2 renders `body`. There is no secondary rich-render bypass.

The default scope is `message`. A `room` or `account` scoped app must include a
stable `app_id`. `account` scope is for global dashboards keyed by the current
Matrix account, not for replacing room-scoped truth.

## Minimal Module Shape

Keep the current shape, but describe it more simply:

```text
src/home/app_registry/
  mod.rs                  envelope parse, registry lookup, fallback boundary
  weather.rs              weather state validation + view model
  news.rs                 news state validation + view model
  templates.rs            include_str table for shipped templates
  templates/<type>/*.splash
  splash_host.rs          static template validation + binding
  widget_manifest.rs      template-reachable widget list
  local_functions.rs      optional render-time helpers
```

The `AppFactory` trait should remain small:

```rust
trait AppFactory {
    fn supported_version(&self) -> u32;
    fn supports_version(&self, version: u32) -> bool;
    fn init(&self, initial_state: &JsonValue) -> Result<Box<dyn RenderedApp>, ValidationError>;
}

trait RenderedApp {
    fn app_type(&self) -> &'static str;
    fn render(&self, language: AppLanguage) -> Result<String, RenderFailure>;
}
```

Do not add `on_action`, `on_tick`, `teardown`, generated-template hooks, or plugin hooks to this trait until a concrete feature needs them.

## Template Rule

V1 templates are static files. They are not generated by the LLM and are not supplied by Matrix events.

Templates may use:

- public Makepad widgets allowed by `widget_manifest`.
- `$state.foo.bar` bindings declared by the app's schema/checker.
- registered local render helpers, if needed.

Templates must not:

- set `capability_id`, `display_name`, `icon`, or `trust_badge`.
- contain arbitrary imperative logic.
- reference unknown widgets.
- reference state paths that the app renderer does not provide.

This is enough for weather/news cards. It avoids turning SplashHost into a general runtime.

## Actions

Robrix2 should not copy aichat's `agent.notify` directly into V1.

In aichat, `agent.notify` works because the app owns:

- the active LLM session.
- local app state.
- the Splash VM.
- the immediate event loop.

Robrix2 does not own the producing agent. It renders Matrix history. If an app needs action buttons, the action should be modeled as a Matrix/OctOS action response, not as local hidden state inside the rendered card.

Therefore:

- The current static V1 cards are display-only.
- Existing OctOS message action buttons remain separate from Splash app cards.
- The first `agent2view` action path may add local reducers for explicitly
  scoped app instances.
- Agent-owned or shared actions should use the existing action-response pipeline.
- A future local-only demo may borrow aichat's `agent.notify`, but that is not the Robrix2 production contract.

## State

There are three separate concepts:

```text
app type      = global definition, e.g. "counter" or "weather"
app instance  = scope + app_id/event_id + current state
view          = one message rendering an app instance
```

An app type is not bound to a message. An app instance is bound to a scope.
One app type may have many instances.

### State Scopes

Robrix2 should support these scopes:

```rust
enum AgentViewScopeKey {
    Message {
        room_id: OwnedRoomId,
        event_id: OwnedEventId,
    },
    Room {
        room_id: OwnedRoomId,
        app_id: String,
    },
    Account {
        account_id: String,
        app_id: String,
    },
}
```

For the first `agent2view` implementation:

- support `message` scope by default.
- support `room` scope when `app_id` is present.
- support `account` scope for explicit global-dashboard apps such as
  `mission_dashboard`.

`message` scope means one Matrix event owns one local app instance:

```text
room_id + event_id -> AgentViewSession
```

`room` scope means multiple messages in the same room may render and mutate the
same local app instance:

```text
room_id + app_id -> AgentViewSession
```

`account` scope means multiple rooms may render the same app instance for the
current Matrix account:

```text
account_id + app_id -> AgentViewSession
```

The first implementation keeps this state in memory. Persistence and conflict
rules are future work.

### AgentViewSession

The first interactive runtime should keep state outside the widget tree:

```rust
struct AgentViewSession {
    scope_key: AgentViewScopeKey,
    source_event_id: OwnedEventId,
    app_type: String,
    version: u32,
    template_id: String,
    state: serde_json::Value,
    dirty: bool,
}
```

Widgets may be destroyed by `PortalList` virtualization. The session map keeps
state alive while the room is open and lets the card re-render when it scrolls
back into view.

### Initial State And Authority

`initial_state` is copied from the original Matrix event content. Edits or
replacements to the visible message body must not rewrite the rendered app. This
matches the current `room_screen.rs` behavior that reads the app envelope from
original event content.

For `message` scope, no persisted client-side app state is needed in the first
phase. For `room` scope, local state may live in memory first; persistence can
be added only after the local reducer model is proven.

If a state change must become shared truth, Robrix2 should send an explicit
Matrix/OctOS action response and let the agent produce a new event. Local
reducers are for immediate view behavior, not for silently rewriting Matrix
history.

### Room Snapshot Projection

For `room` and `account` scoped apps, Matrix events are full snapshots. When
RoomScreen sees a valid app snapshot, it binds it to the scoped
`AgentViewSession`.

Projection rules:

- a later timeline item for the same scope replaces the session state.
- an older timeline item redrawn by `PortalList` must not roll the session back.
- a redraw of the same event must not clear local `dirty` reducer state.
- a new producer snapshot clears `dirty`, because shared Matrix state has
  advanced.

This keeps room-scoped apps stable under virtualization while preserving the
Matrix timeline as the source of shared truth.

### Local Reducers

The first `agent2view` reducer model should be narrow and app-type specific:

```text
counter.inc
counter.dec
counter.reset
timer.toggle
timer.reset
```

The reducer lookup should be keyed by app type and action id, not by an
unbounded global `agent.notify("anything")` string. Unknown actions are dropped
or routed to the action-response path.

The click loop is:

```text
button action
  -> action carries scope key + action_id + payload
  -> lookup AgentViewSession
  -> run whitelisted local reducer
  -> re-render the affected view(s)
```

For `message` scope, re-render one card. For `room` scope, re-render all visible
cards that point at the same `room_id + app_id`.

## Safety Model

The v1 safety argument should be short:

1. Rich UI is rendered only for registered app types.
2. Each app validates its own `initial_state`.
3. Templates are shipped with Robrix2, not generated at runtime.
4. SplashHost rejects templates that reference disallowed widgets, disallowed helpers, undeclared state paths, or host-owned attribution fields.
5. Any failure falls back to plain text.

This replaces the current seven-whitelist framing for V1. The detailed whitelist taxonomy can remain as future design notes, but it should not be the main explanation for the shipped path.

## Relationship To Aichat

Aichat proves a separate runtime-agent pattern:

```text
runsplash button
  -> agent.notify(event_id, payload)
  -> host bare action
  -> host state mutation or LLM prompt
  -> markdown/Splash re-render
```

That is useful reference material for future Robrix2 L2 actions, but Robrix2 v1 should stay event-sourced:

```text
agent-produced Matrix event
  -> static template render
  -> fallback body on any failure
```

The core lesson from aichat is not "use agent.notify everywhere." The core lesson is to keep the first live wire narrow and prove it before adding generalized runtime machinery.

## Migration From Current Design

1. Treat `docs/design/agent-to-app-design.md` as an aspirational/full architecture note, not the current implementation contract.
2. Use this simplified design as the v1 contract for Robrix2.
3. Rename future work into explicit follow-up tracks:
   - L2 actions: card buttons routed through Matrix/OctOS action responses.
   - L3 stateful runtime: timers, local state, ticks, and replay semantics.
   - Generated templates: opt-in, with static fallback and validation.
4. Keep weather/news as the proof that new display cards need no per-card Rust UI string builder.
5. Do not add more framework surface until a third display-only capability or a concrete action-capable card forces it.

## Success Criteria

The simplified v1 design is working when:

- Weather and news render through static Splash templates.
- Invalid app envelopes fall back to body.
- Unsafe templates fail host preflight and produce no Splash output.
- The first `agent2view` runtime can distinguish app type, app instance, and view.
- `message` scoped app instances are isolated by `room_id + event_id`.
- `room` scoped app instances are shared by `room_id + app_id`.
- Adding a new display-only capability requires only state validation, a view model, and one or more static `.splash` templates.
- No v1 code path depends on LLM-generated Splash, generated template repair, dynamic plugin discovery, or client-side persistent app state.
